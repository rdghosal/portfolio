searchBar.js

A Kintone plugin for approximate string matching

Context

  • A database application with over 700 records on Kintone, an online cloud platform

Problem

  • Strings of Latin characters require the entire string as a query, making querying inefficient and inconvenient
  • Neglecting this issue could lead to disuse and abandonment of the platform—a potential loss of over $2000 in licensing costs A database application with over 700 records on Kintone, an online cloud platform

Specifications

  • Allows impartial string queries to match intended targets
  • At least as fast and convenient as built-in search features to merit adoption
  • Distinguishes between string and number (JavaScript) data types to allow “fuzzy” matching for string queries and ensure precision for numerical queries
  • Queries all records, despite pagination in application view
  • Outputs search results such that matched records have only relevant information displayed while being aesthetically pleasing Queries all records, despite pagination in application view

Technologies

  • JavaScript ES6
  • Bootstrap

Architecture and algorithm design

  • Use native JavaScript methods for string matching to meet specifications and time constraints, while avoiding overhead in design and testing for more precise fuzzing matching algorithms that use, for example, edit distance
  • Implement conditional logic that branches to different iterations of the designed filtering algorithm according to whether the data is of type string, number, or object (in case of a nested-table)
  • Fetch all records prior to execution of the filtering algorithm through an asynchronous HTTP request to the Kintone API in order to optimize execution time
  • Output results as a separate tab to which the browser automatically navigates to and styling such using Bootstrap

Testing

  • Manual testing using Google Chrome v79
  • While record loading is susceptible to latency, the filtering algorithm meets specifications while appearing to be no slower than built-in search features

SpyceGen.py

A console application to automate the generation of scripts used in HSPICE® printed circuit board (PCB) signal and power integrity simulations

Context

  • PCB simulations achieved by several desktop applications, one of which is Synopsys® HSPICE®

Problem

  • Automating the generation of execution scripts for HSPICE®

Specifications

  • Replaces the manual and error-prone method of copy-pasting boilerplate code and string data found in files external to the script file
  • Generates a script file and permutations thereof to satisfy simulation requirements (e.g., a fast, slow, and typical case per interface signal)

Technologies

  • Python 3.7 standard library
  • Pytest

Architecture and algorithm design

  • Assume a pre-determined directory structure used to navigate and extract data
  • Reference a template file containing boilerplate code
  • Use regular expressions to parse folder/file names and in-file string data
  • Represent sections of the script file as objects with behavior to mutate string data according to patterns within those sections
  • Implement hash tables (Python dictionaries) to store data in heap memory for objects to efficiently access at runtime

Testing

  • Unit testing using Pytest
  • Testing parameters reflect a particular use case, thus limiting the application’s usability

Weaver.py

A console application to automate the generation of reports made in PowerPoint

Context

  • A reporting service provided to clients wherein reports are generated in PowerPoint

Problem

  • Automating the generation of PowerPoint reports of varying type

Specifications

  • Generates a PowerPoint file with data consistent with that of an initial report delivered to clients that confirm service conditions (hereafter, “confirmation report”)
  • Does not alter any referenced files (e.g., confirmation report)
  • Generates a file and permutations thereof according to the report’s type

Technologies

  • Python 3.7 standard library
  • pywin32
  • Pytest

Architecture and algorithm design

  • Utilize pywin32 over IronPython, Python for .NET, and python-pptx to interface PowerPoint through the Microsoft Component Object Model (COM), lending the application to faster development and greater usability, maintainability and reliability
  • Reference a template file for each report type containing patterns in file structure and content
  • Represent each report type as an object with specific behavior, which ensures data writing according to file structure, achieved through polymorphism
  • Represent confirmation tools as an object with data reading behavior to prevent changes to file
  • Implement algorithms to query Slide objects for specific shapes, such as Tables and TextFrames consistently
  • Modularize objects and constant variable definitions to ensure usability in future applications

Testing

  • Parametrized unit testing using Pytest
  • End-to-end testing using reports of all types from an actual use case

Wave Native

React / Flask Web Application

Background

Living in Japan, I was often met with English learners, who had a proficient grasp of the language, but often struggled with pronunciation of certain words. This inspired the idea of Wave Native, an application dedicated to serious students of the language with the following features to assist their learning:

  1. Audio files for queried words that could be played on the browser or downloaded.
  2. A recording interface whereby users can attempt their own pronunciation of words.
  3. Visual representations of fetched audio files and user recordings, such that the user may compare their attempt against native pronunciation.
  4. User profiles where a user may track their commonly queried words.

My Role

In this project, I was the sole developer and project owner. Thus, my responsibilities comprised of the following:

  1. Understanding the domain of language acquisition, particularly as is related to pronunciation.
  2. Selecting the choice of visual representation for auditory data, with a focus on the question of "What would be most relevant to users without backgrounds in linguistics and other technical fields?"
  3. Finding an accessible source for audio data, as supplying the data myself was infeasible.
  4. Architecting and developing both the client and API.
  5. Architecting the database and designing the necessary SQL queries.
  6. Testing the application, namely though manual end-to-end testing.
  7. Deploying the application.

Technologies

In this project, I had chosen the following tech stack.

  1. React

    2. In the design of the client, there were elements whose styling and behavior would need to be consistent throughout. As such, React, which is modeled on the idea of components, appeared to be an appropriate choice.

  2. Flask

    3. Anticipating a need for computation and numerical analysis, I chose Python as the language for the backend. Without a need for a number of the features included in Django, Flask appeared to be an appropriate choice as a lightweight API framework.

  3. Postgresql

    4. The data models lends itself to a relational database, and Postgresql provides a free-tier fitting the project budget.

Challenges

In the planning and development of this project were the following challenges.

  1. Back-end performance

    2. Given the necessity to source data from a third-party API, each query made in the client will require at least two network calls: (1) Between the client and the Wave Native API and (2) Between the Wave Native API and the third-party API. Network calls are expensive from a performance perspective and could prove especially so for users with poor internet connection.

  2. Algorithm design

    3. Features wherein the user has more quantitative feedback would be useful, especially in terms of tracking language acquisition progress. For example, an algorithm wherein native and user-recorded audio files are compared could provide helpful feedback to the user. Implementing this, however, requires a greater level of subject matter expertise, particularly in audio file comparison that accounts for situations where the user is not in ideal settings for recording and other circumstances that could negatively impact accuracy of the algorithm.

  3. Browser support

    4. The recording interface (whereby the user may recording their own attempt at pronouncing a word), faces the issue of browser support namely as the browser API for microphone access varies. Therefore, while microphone access might be readily available in Google Chrome, in other browsers, like Safari, this may not be the case and, as a result, a major feature could be rendered inaccessible to a subset of users. Resolving this issue requires study and implementation of workarounds for browsers that are currently incompatible.

To Do: Points For Improvement

In retrospect, there are several aspects of the application that I hope to improve in the future, both at a code and design level.

  1. Input validation

    2. Currently, input data is not sanitized. That is, the application is vulnerable to attempts at hacking by the user and requires more robust security. Possible solutions here could include the inclusion of a third-party library that, for example, checks user input for attempts at SQL injection attacks and the like.

  2. UI design

    3. While the user may be able to intuit usage of the application from the UI, the UX could be better served with more intentional visual hierarchy. For example, the media buttons, whereby the user interacts with the audio file, break from the overall color scheme of the site while not being consistent with each other, thereby obscuring the call-to-action. A similar comment can be made for the user login modal, which has, again, different colored buttons for no particular reason from a UX-perspective, while also violating good design principles, such as negative space, with its lack of padding.

  3. Front-end architecture

    4. Although the front-end is divided into React components, the current segregations of state-related logic are not intuitive. State, for instance, is often being affected by components that should not have such authority. As such, the client code is susceptible to bugs and could be improved with an architecture oriented toward the flow of data in the client.

  4. Back-end architecture

    5. From a code maintenance perspective, the API is separated into controller, model, and service layers, which limits coupling between logic that is inherently separate. However, closer examination of the code reveals that the Single-Responsibility Principle is violated in places, where, for example, service layer logic is in a controller and the like. Such instances are worth refactoring, particularly if there are plans to expand or enchance the feature set of the API.
